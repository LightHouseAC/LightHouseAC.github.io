一、设计模式（主要学单例模式，工厂模式Factory，建造者模式Builder，代理模式Proxy）

七大原则：

设计原则 定义
开闭原则 对扩展开放，对修改关闭
单一职责原则 一个类只负责一个相应职责
里氏替换原则 所有引用基类的地方必须能透明地使用其子类的对象
依赖倒置原则 依赖于抽象，不能依赖于具体实现
接口隔离原则 类之间的依赖关系建立在最小的接口上
合成/聚合复用原则 尽量使用合成/聚合，而不是通过继承来复用
迪米特法则 一个软件实体应尽可能少地与其他实体发生相互作用

设计模式的分类：

  • 创建型：创建对象的同时隐藏逻辑，不使用new直接实例化对象（Major：工厂/抽象工厂/单例/建造者模式）！！！重点！！！
  • 结构型：通过类和接口见的继承和引用实现创建复杂结构的对象（Major：适配器/代理模式）
  • 行为型：通过类之间不同通信方式实现不同行为（Major：观察者模式）

工厂模式

  • 简单工厂模式：由一个工厂对象来创建实例，客户不需要关心创建逻辑，只需要传入工厂的参数

​ Spring中：BeanFactory使用简单工厂模式，根据传入一个唯一标识获得Bean对象

  • 工厂方法模式：简单工厂模式中工厂负责生产所有产品，而工厂方法模式将生产具体产品的任务分发给具体的产品工厂

    定义一个抽象工厂，定义产品的生产接口，不负责具体的产品。不需要通过指定类型来创建对象，生产任务会分发出去

  • 抽象工厂模式：抽象工厂模式在抽象工厂中增加 创建产品 的接口，在具体的子工厂中实现新加产品的创建

单例模式

一个单例类在任何情况下都只存在一个实例，构造方法是**(1)私有的，由自己创建一个(2)静态变量存储实例，对外提供一个(3)静态公有方法**获取实例。

优点就是内存里面只存一个实例，减少内存开销。针对大型、复杂或者频繁创建和销毁的实例，使用单例模式可以大大减少内存开销。

缺点就是没有抽象层，难以扩展。

  • 单例模式常见实现方法

    1. 饿汉式 - 线程安全

      • 类一加载就创建对象，比较常用但是容易产生垃圾对象，不是懒加载

      懒加载Lazy Loading：使用的时候再创建对象

      • 怎么保证线程安全？基于类加载机制避免了多线程的同步问题

      通过反射是可以获取私有对象的，因此通过反射就可以获得这个构造器访问权，因此反射可以破坏单例模式

      饿汉式实现的单例模式可以防止被反射破坏

      懒汉式 实现的单例模式无法防止被反射破坏

    2. 懒汉式 - 线程不安全
      单线程下可以这样实现，但是多线程无法保证单例（需要加锁，见方法3）

      这样的实现显然是线程不安全的，new出来的Thread随便创建不同的实例，破坏单例

      用前面的方法来防止反射破坏也是做不到的，只需要

      1. 先用反射创建对象，这时候instance != null判断是无效的，创建了一个不一样的实例
      2. 再正常创建单例，又是不一样的实例

    3. 懒汉式 - 线程安全（加锁）

      虽然懒汉式防止不了反射破坏单例，但是可以防止多线程破坏，就是通过加锁的方式

      但是 每次调用getInstance 获取实例对象都要加锁和释放锁，消耗性能

    4. 双重检查锁（双检锁DCL）- 需要volatile关键字

      • 实现原理：

        1. 正常判断，非空说明已经有实例，返回即可，没有才需要判断

        2. 多个线程进行锁竞争，第一个拿到锁的发现没有实例，创建后释放锁，其他线程再进来就发现已经有实例了，返回单例

        3. 这里需要用volatile关键字是因为 new 关键字不是原子操作，编译器可能会把指令重排序

          new一个对象的步骤是

          1. 堆内存开辟内存空间
          2. 调用构造方法，初始化对象
          3. 引用变量指向堆内存空间

          要防止重排序导致的初始化后导致应该123顺序执行的变成了132执行，导致使用了没有初始化的，非空的实例（判断instance == null的时候没有初始化）

    5. 静态内部类 （最简单的安全实现方式）

      静态内部类不属于 5种 类的主动引用，属于被动引用

      当getInstance方法被调用时，InnerClass才会在Singleton的运行时常量池里面，这时候INSTANCE才被真正创建并返回出去

      至于保证线程安全则是 JVM 保证了类的class init：<clinit>()方法在多线程时被正确加锁和同步

    6. 枚举单例 通过枚举实现单例

代理模式

代理模式的本质是一个中间件，主要目的是解耦服务提供者和使用者（好像中间件干这活的挺多的）

3个概念：

  1. RealSubject：真正的目标对象
  2. Subject：定义了RealSubject对外的接口，这些接口必须被Proxy实现，这样调用Proxy的接口最终都转化成调用RealSubject
  3. Proxy：目标对象的代理，控制和管理目标对象，并且间接地传递外部对目标对象的访问
    1. Remote Proxy远程代理：对本地的请求以及参数序列化，向远程对象发送请求，将响应结果反序列化后返回给调用者
    2. Virtual Proxy虚拟代理：目标对象的创建开销比较大的时候，用延迟或者异步的方式创建目标对象
    3. Protection Proxy保护代理：细化对目标对象访问权限的控制
  • 静态代理和动态代理的区别
    • 动态代理更灵活，不需要必须实现接口，可以直接代理实现类，不用对每个目标类都创建代理类
    • 静态代理在编译时就将接口、实现类、代理类都变成了class文件。动态代理在运行时动态生成字节码加载到 JVM里

字节码：.class文件，JVM可以理解的代码

只面向虚拟机，通过字节码实现了可移植，性能比一般解释型语言高，由于只面向虚拟机，Java程序无需重新编译即可跨平台运行

.java -> javac编译 -> .class -> 解释器/JIT -> 机器可以理解的代码

JIT编译器：Just in Time Compilation，运行时编译，用于编译热点代码（编译速度>>解释）。完成编译后将字节码对应的机器码保存下来。

因此Java是 编译与解释共存的 语言

JVM里面有一个加载类的子系统，运行时区域（方法区域，堆，栈，线程池，原生方法栈），原生方法库，原生方法接口，和执行引擎

Java的基本类型和包装类型
  1. 基本类型一般只用于定义一些常量或者局部变量。包装类型可以用于泛型，但是局部类型不可以
  2. 基本类型的局部变量存在 JVM的栈中，成员变量存在 JVM的堆中，static修饰的成员变量属于类不属于对象，也存在堆里。包装类型属于对象类型，几乎所有对象实例都存在堆中
  3. 基本类型占用的空间更小，基本类型有默认值非null，但是成员变量包装类型不赋值就是null
  4. 基本数据类型 用 == 比较的是值，包装类型 用 == 比较的是对象的内存地址，包装类型比较值需要用equals()
包装类型的缓存机制

[-128,127], Character[0,127]范围内缓存了数值范围内对应的对象

因此包装类型的对象，如Integer可能在IntegerCache.cache产生，复用了已有对象，范围外的对象，会在堆上产生，不会复用

不仅如此，new出来的对象用 == 也是判断为false的，缓存范围内数值和valueOf的才会被 == 判断为true

所以一定要用equals方法判断包装类型的数据

静态变量/方法

static 静态变量，被类的所有实例共享，只会被分配一次内存

static被final修饰变为常量

静态的方法不能调用非静态的成员，静态方法是属于类的，非静态成员属于实例对象，对象实例化后才存在

调用静态方法无需创建对象 => 比如说utils类，如RandomUtil.randInt()，可以直接类名.方法名调用

而实例方法需要实例来调用，如User user = new User(); user.setEmail("a@b.com");

重载&重写

重载即同一个方法根据不同参数做出不同的处理

重写则是子类继承父类的相同方法/参数，做出不同的处理，打@Override注解，发生在运行期

不能重写构造方法和父类被 private/final/static 修饰的方法，但是可以重新声明static方法（父类写了一个toString不影响子类也写一个）

OOP相关

面向对象方法一类编程思想，其核心是把一类问题抽象出对象，然后调用对象的方法来解决问题

new的对象实例存在堆内存中，对象的引用存在栈内存中。引用可以看成一个地址名片，知道在哪可以找到想要访问的对象

对于引用数据：== 比较的是内存地址， equals原本就是一个 == 判断（，但是String类重写了该方法，改为判断值是否相等）

对于基本数据： == 比较的是值，基本数据不属于Object类，没有方法，自然也没有equals方法

封装，继承和多态

封装即属性私有，但提供公开方法供调用(Getter/Setter)

继承extends，子类拥有父类所有方法，但是子类本质上也是一个新的类，只不过可以沿用或重写父类方法，比较方便且实现了代码复用

相当于父类作为一个模板，子类可以开箱即用，也可以在其基础上做扩展

多态：父类的引用指向子类的实例，如

其中，ArrayList和LinkedList都继承了AbstractList，父类的引用strList可以指向new出来的子类的实例

多态的 对象 和 引用 之间可以是继承(类class) 和 实现(接口interface)，多态不能调用子类存在但父类不存在的方法，但是如果调用被重写了的方法，执行的是子类的方法

接口和抽象类：接口用于管理和约束类的行为，实现一个接口就具有了规定的行为（需要实现接口定义的所有方法），而抽象类主要用于代码复用，你可以继承来一个方法直接用，也可以去重写它

一个类只能继承一个类，可以实现多个接口：public class A extends B implements C,D,E,F,G...

Object类

所有类的父类，原生提供了一些方法：getClass, hashCode, equals, clone, toString, notify, notifyAll, wait, finalize

hashCode()：获取Hash码，用于确定对象在哈希表中的索引位置，这样往Map、Set里存对象的效率就更高了

hashCode相等不一定对象就相等，存在哈希碰撞。但是对象相等hashCode一定相等

因此重写equals方法一定要重写hashCode方法，否则可能导致HashMap里判断2个对象的hashCode不同从而把相同对象存入了不同位置

String类

String是不可变的，可以当成常量（会存在字符串池里面，引用相同值的字符串会指向同一个地址，除非new了一个String对象）

String的不可变是通过 (1) private + final修饰它的底层保存变量（Java 8及以下char数组，Java 9及以后byte数组）—— 不可被访问不可被继承，而且 (2) 不提供公共方法修改 来实现的。在Java8及以下的版本中，反射可以破坏这种不可变的特性（毕竟private对反射无效，final又只在编译期生效），高版本也有一些绕过保护的方法

StringBuilder

Java8及以下在循环中用 + 拼接字符串会创建大量StringBuilder对象，最好使用 StringBuilder 的 append 方法来拼接

字符串常量池

JVM为了避免字符串重复创建而开辟的一块区域，如果字符串对象已经在字符池里面了，就直接返回引用，否则创建一个并把引用保存至池子

String s = new String("str");会创建 1 或 2 个字符串对象。

  1. 若池子里没有字符串对象”str”的引用，则会在堆上创建2个字符串对象，其中一个的引用会被保存在字符串池里

    堆中先创建1个空对象SO1，再创建一个”str”对象SO2并保存引用到常量池，然后给前面创建的对象赋值并返回SO1的引用

  2. 若池子里有引用，则只会创建一个对象

字符串intern()方法

str.intern() 方法会将指定字符串对象的引用保存到常量池中

  1. 若常量池中保存了对应字符串对象的引用，就返回引用
  2. 若没保存引用，就创建一个引用并返回

一般用于给String对象上锁，见 Blog #4

String类型的变量与常量 + 运算

在编译期，编译器就会把字符串常量给存到字符池里面，而含有变量的则不会，不过用final声明的字符串所在的表达式也会被编译器优化

例如：

常量折叠只会优化

  2. final修饰的 基本数据类型/字符串 变量

异常

java异常类主要分为 Exception 和 Error 都继承自 Throwable （所有异常的祖先）

  • Exception: 程序可以处理的异常，可以被 catch 捕获，分为 Checked Exception 和 Unchecked Exception
    • Checked Exception: 受检查异常，必须处理。如果没被 catch 或 throws 处理则无法通过编译
    • Unchecked Exception: 不受检查异常，可不处理 —— Runtime Exception及其子类
  • Error: 程序不能处理的错误，JVM遇到这类异常一般会选择终止线程
泛型

泛型可以增强代码的可读性和扩展性（可以兼容很多传递的类型），就是 <T> 这个东东，最常用的比如 List<Integer> aList = new ArrayList<>();

编译器可以对泛型参数进行检测，通过指定泛型参数可以指定传入的对象类型，比如上面的 Integer

反射

反射算是一个优缺点鲜明的机制，经常听到反射破坏xxx，其实就是因为反射机制的能力太高了，可以获取任意一个类的所有属性和方法

但是反射用起来非常灵活，比如常见框架的 动态代理、注解等功能 就是通过反射实现的

注解

注解用于修饰类、方法或者变量，可以理解成给程序打的带有信息的注释

注解有编译期直接扫描时生效的，也有运行期通过反射来处理的（如Spring框架里的@Value, @Component等）

SPI

Service Provider Interface，给服务提供者或扩展框架功能的开发者使用的接口，解耦服务的调用方和服务的提供者，可以提升程序的扩展性、可维护性，修改或替换服务实现不需要修改调用方。框架想要有更多的扩展能力就会使用SPI机制

SPI和API的区别是，API是提供方提供好了接口和实现，直接调用即可。而SPI是接口存在调用者这边（调用者定义好接口标准），提供者提供不同的实现来提供服务

比如我实现了一个RPC框架，我可以依赖SPI机制，来实现只需要在META-INF中自定义一个实现，就可以后续调用这个实现

Java本身提供了一个 ServiceLoader 工具类来加载服务实现，也可以自己写一个 SPILoader 扫描路径并加载实现类

SPI机制能提高扩展性和灵活性，但是SPI的效率比较低

序列化/反序列化

Serialize序列化，对象=>二进制字节流，反序列化相反。

序列化常见场景：对象通过网络传输（如RPC），对象存储到文件，对象存储到数据库（如Redis），对象存储到内存

属于TCP/IP 4层模型的应用层，属于OSI七层模型的表示层

不想序列化的变量可以用transient修饰（反序列化后值会变为默认值）。static变量不属于对象，不会被序列化

JDK序列化器效率低，存在安全问题且不支持跨语言调用，一般不用。用Hessian, Kryo, Json（为了可读性牺牲效率时用）

2) Java集合

Java集合（容器）由 Collection 和 Map 接口派生而来

  • Collection 接口：

    主要存放单一元素（List, Set, Queue等）

    • List
      • ArrayList: Object[] 数组
      • Vector: 线程安全但效率较低的Object[] 数组
      • LinkedList: 双向链表
    • Set
      • HashSet：无序集合
      • LinkedHashSet：链接的哈希集合，有容量和负载因子
      • TreeSet: 有序集合（红黑树，同C++的Set）
    • Queue
      • PriorityQueue: 优先队列
      • Deque: 双端队列

  • Map 接口：

    主要存放K/V键值对

    • HashMap: 数组 + 链表实现（链表散列），链表是为了用 “拉链法” 解决哈希冲突。JDK1.8以后当链表长度大于阈值时将链表转化为红黑树减小搜索时间

      • HashMap的hash方法：

        HashMap数组的每一项都是一个链表，如果产生了哈希冲突，就把冲突的值放进链表里（当链表长度>阈值，且数组大小>64时，JDK1.8以上版本会将链表转变为红黑树）

      • HashMap的长度为什么都是2的幂次

        因为Hash映射的空间特别广，需要把它对数组的长度取模后才能用来当存放的位置（数组下标）

        但是%运算比较慢，性能不好，用到数论里面，p对2的幂次取模 p%x 等价于 p&(x-1)， 可以大大提升效率

    • ConcurrentHashMap: 保证线程安全的HashMap

      • HashMap为什么线程不安全？

        因为多个线程进行put操作产生哈希冲突时时候可能导致数据覆盖

        线程A检查哈希冲突后还没插，如果这时CPU把资源分配给了线程B，线程B检查冲突然后把数值插了进去

        然后线程A继续执行插入，就覆盖了线程B插入的数据

      • ConcurrentHashMap则会对链表的一个节点上锁，保证线程安全的同时提升了并发效率，不会说一个线程put，另一个就不能put了，（对比Hashtable只用一把锁全锁了的方式，ConcurrentHashMap的锁粒度更细）

      • 线程安全不意味着保证了复合操作的原子性（数据覆盖还是会产生），要保证原子性，还需要靠ConcurrentHashMap提供的一些原子性操作

        如：map.putIfAbsent(k,v);

    • LinkedHashMap

    • Hashtable

    • TreeMap: 红黑树

3) Java并发

Java线程

线程是比进程更小的执行单位，一个进程执行的过程中可以产生多个线程。线程共享进程的 堆 和 方法区 资源，但是拥有自己独立的 程序计数器，虚拟机栈和方法栈 资源。操作系统切换线程上下文或者创建线程时的开销要小很多。

Java的线程是基于原生线程实现的，JVM直接调用内核线程来实现Java线程，由操作系统内核来进行线程的调度和管理

也就是在线程的3种映射模型中，Java使用的是1对1模型

线程模型：分为多对一模型，多对多模型和一对一模型，反映用户线程映射到内核线程的情况

用户线程：由用户态程序管理和调度的线程，运行在用户态，切换和创建成本低，不能利用CPU的多核

内核线程：由操作系统内核管理和调度的线程，运行在内核态，切换和创建成本高，但是可以利用CPU的多核

  1. 程序计数器、虚拟机栈和方法栈私有

    • 程序计数器：需要记录线程执行的位置（循环，分支控制等情况），以及多线程被切换回来后才知道从哪继续，所以需要私有
    • 虚拟机栈和方法栈：Java方法的调用对应栈帧（存放局部变量等信息）在栈中出入，需要私有来保证线程中的局部变量不被别的线程访问

  2. 堆和方法区

    堆主要存放新创建的对象（几乎所有对象都在堆内分配内存），而方法区主要存放已加载的类信息，常量、静态的变量等

线程的生命周期

初始状态（刚创建），可运行状态（正在运行或者等CPU时间片），终止状态（结束执行）和休眠状态（阻塞、等待）

线程上下文切换

线程的切换时，需要保存线程的上下文，然后腾出CPU给其他线程用，下次再占用时恢复现场

线程的 sleep 和 wait

sleep方法没有释放锁，wait会释放锁

wait不设置timeout不会自己苏醒，需要其他线程调用同一个对象上的 notify 或者 notifyAll

所以wait一般用于线程间交互/通信，sleep用于暂停执行

sleep方法属于Thread，wait方法属于Object

因为sleep是让当前线程暂停执行，不涉及到对象类，也不需要获得对象锁

而wait是 用于线程间交互/通信的 ：线程只需要知道知道某个对象是否占有锁，wait 会释放锁，把 wait 方法放在Object可以让任何对象拥有实现线程通信机制的能力

直接调用Thread类的run方法

new一个Thread对象，线程会进入新建状态。调用start方法，会启动一个线程并使线程进入就绪状态，分配到时间片就会进入运行状态了，start方法会执行线程的相应准备工作，然后自动执行run方法。

直接执行run方法则会把它当成main线程下一个普通方法执行，而不是在某个线程中执行

因此调用start会启动线程并使线程进入就绪状态，直接执行run方法不会以多线程的方式执行

Volatile关键字

可以保证变量的可见性：告诉JVM，这个变量共享且不稳定，每次使用它都到主存中读取

可以禁止JVM的指令重排序（应用场景：双检锁实现单例模式）

不能保证对变量操作的原子性

乐观锁和悲观锁

悲观锁假设最坏的情况，认为每次共享的资源被访问都会出问题，所以每次获取资源操作前都会上锁。

共享资源每次只会给一个线程使用，其他线程会阻塞。如：synchronized 和 ReentrantLock 等

高并发场景下，激烈的锁竞争会造成大量的线程阻塞，导致频繁的上下文切换，增加很多系统开销

而乐观锁总是假设最好的情况，认为每次访问都不会出现问题，只有在提交修改的时候取验证是否被其他线程修改了

一般情况下，悲观锁适用于写比较多的场景，避免乐观锁频繁失败重试影响性能，而乐观锁适用于读比较多的场景，避免频繁上锁影响性能

乐观锁一般用版本号机制和CAS算法来实现

版本号机制：给数据加上一个版本号字段，表示修改次数，当数据被修改时+1，提交修改时要校验刚刚读取出来时的版本号和库中的是否一致

synchronized关键字

synchronized关键字可以修饰 实例方法（锁当前对象实例），静态方法（锁当前Class类）和代码块（锁指定的对象/类）

synchronized 对加锁会获得一个监视器monitor，只有拥有监视器的线程才可以释放锁

ReentrantLock

也是可重入而且独占式的锁，和synchronized类似，但是更灵活，加了一些高级功能

公平锁和非公平锁

公平锁指的是锁被释放后先申请的线程先得到锁，性能较差；而非公平锁则是随机或者按某种方式给的锁，性能更好，但是可能导致某些线程永远无法获取到锁

可重入锁：线程可以再次获取自己的内部锁。线程A获得了一把锁，如果它还没释放，还想继续获取这个锁也是可以的

一般锁和读写锁的区别：

  • 一般锁：读读互斥、读写互斥、写写互斥
  • 读写锁：读读不互斥，其他互斥

共享锁和独占锁

  • 共享锁可以被多个线程同时获得
  • 独占锁只能被一个线程获得

有读锁不能获取写锁，有写锁可以获取读锁：因为获取写锁需要没有占用读锁，而获取读锁时只有发现写锁被别人占了才获取失败

写锁可以降级成读锁，但是读锁不能升级成写锁

读锁升级成写锁会引起线程的争夺会影响性能，也可能造成死锁：2个想升级的线程都等着对方释放自己的读锁，然后都不释放

ThreadLocal

一般情况下变量可以被所有线程访问，如果想实现每个线程都有自己专属的本地变量，就需要ThreadLocal

ThreadLocal就是每个线程都可以有的一个自己的小包，可以往里面塞自己线程的私有数据

线程池

池化技术的思想就是减少创建和销毁的开销，提高资源利用率

每当有任务要处理时，从线程池获取线程来处理，处理完后线程回到池子里等下一个任务，而不是销毁

线程池不仅可以 降低资源消耗， 提高响应速度（不需要重新创建），还可以提高线程的可管理性

Future类

Future类是异步思想的典型运用，主要用于耗时操作

可以把耗时操作交给 Future 去做，然后去做其他事情，最后再从 Future 那里拿回来执行结果

CompletableFuture

CompletableFuture解决了Future类的一些局限性，提供了函数式编程和异步任务编排组合等能力

异步编排组合：将多个异步任务串联起来，组成一个完整的链式调用

为什么网络要分层

复杂的系统需要分层，每一层专注地完成自己的事情

网络分层可以让各层之间相互独立，提高灵活性，大问题化小

经典输入URL到页面展示全流程
  1. 浏览器检查DNS缓存，发现没有就去DNS服务器获取，得到URL对应的IP地址和端口号
  2. 路由器向web服务器发送一个TCP连接请求，完成三次握手后，建立起请求
  3. 浏览器通过TCP连接向服务器发送获取页面的HTTP请求报文
  4. 服务器收到请求后处理，返回HTTP响应报文
  5. 浏览器收到响应后解析响应体的HTML代码，如果发现代码中还有其他要请求的资源，则继续发送请求获取资源直到页面加载显示完成
  6. 完成后浏览器可以主动关闭TCP连接或者等待服务器关闭
HTTP状态码
状态码 类别 代表信息
1xx 信息性状态码 接收的请求正在处理
2xx 成功状态码 请求正常处理完毕
3xx 重定向状态码 需要进行附加操作完成请求
4xx 客户端错误状态码 服务器无法处理请求
5xx 服务端错误状态码 服务器处理请求出错
常见状态码 含义
200 OK 请求被成功处理
201 Created 请求被成功处理并在服务端创建了一个新的资源
202 Accepted 服务端接收到了请求，但是还没有处理
204 No Content 服务端成功处理了请求，但是没有返回内容（用于只关注请求是否被成功处理）
301 Moved Permanently 资源被永久重定向了
302 Found 资源被临时重定向了
400 Bad Request 请求错误（可能是参数或者方法）
401 Unauthorized 未认证
403 Forbidden 直接拒绝不处理（用于非法请求）
404 Not Found 请求的资源在服务端不存在
500 Internal Server Error 服务端出问题了（bug或其他原因）
502 Bad Gateway 网关将请求转发到服务端，但是返回了错误的响应

HTTPS

保密性好，信任度高

核心——TLS协议

建立连接
  1. 客户端发请求建立连接
  2. 服务器把数字证书和加密算法发回给客户端
  3. 客户端验证证书合法后，1) 产生一个随机密钥，2) 计算握手信息的摘要 3) 用密钥将摘要加密 4) 用证书里的公钥加密密钥。然后把加密后的握手信息和密钥发给服务器
  4. 服务器收到加密后的密钥后用自己的私钥解密，得到先前产生的随机密钥，用改密钥对摘要解密，再自己加密摘要做比对，如果一致就握手成功，后面的通信数据都用这个随机密钥来加密

核心就是保证这个随机产生的密钥安全地到达服务器手里，同时客户端也可以信任它的通信对象是服务器

HTTP1.0 vs HTTP1.1
  1. 1.1 新添加了很多状态码
  2. 1.1默认改用长连接（一次TCP连接可以反复发HTTP请求）
  3. 1.1添加的host字段可以让服务端确定客户端真正想要请求的网址
  4. 1.1新增的100状态码可以用来询问服务器是否愿意响应某些请求
  5. 1.1引入了 cache-control 来控制缓存
HTTP1.1 vs HTTP2.0
  1. 2.0引入了多路复用：在同一个连接上可以同时传输多个请求和响应，互不干扰（1.1是串行方式，每个请求和响应都需要独立的连接）【长连接是可以反复发，多路复用是可以反复发 + 可以发多个】
  2. 2.0使用二进制帧进行数据传输，1.1则是文本格式的报文
  3. 头部压缩：1.1支持body压缩，不支持header压缩。而2.0支持了header的压缩
  4. 服务器推送：2.0支持服务器推送，客户端请求一个资源时，把其他资源一并推送给客户端，减少客户端的请求次数和延迟
HTTP2.0 vs HTTP3.0

  1. 传输协议：HTTP2.0及以前是基于TCP协议实现的，HTTP3.0则是基于UDP升级的QUIC协议。

    QUIC协议： （使用了TLS1.3的特性，支持1个RTT的握手和0个RTT的握手），连接建立仅需0或者1个RTT的额外时间

  2. HTTP3.0一定程度上解决了队头阻塞的问题

    队头阻塞：由于多路复用，多个请求使用1个TCP连接，一旦发生丢包就会阻塞住所有的HTTP请求

    HTTP3.0在一个连接建立了多个不同的数据流，这些数据流之间独立且互不影响

  3. HTTP3.0有更好的错误恢复机制。2.0的依赖于TCP的错误恢复机制

  4. HTTP2.0使用TLS协议加密，HTTP3.0基于的QUIC协议包含内置的加密和身份验证机制（TLS1.3）

事务

事务的特性：ACID，原子性，一致性，隔离性，持久性

原子性通过undo log保证，隔离性通过MVCC和锁机制保证，持久性通过redo log保证

并行事务导致的问题

  1. 脏读

    一个事务读到了另一个未提交事务修改过的数据

    事务A读到了事务B改的数据，但是事务B回滚了，这时候A就读到了过期数据

  2. 不可重复读

    一个事务内多次读取同一个数据，前后读到的数据不一致

    事务A读一次，然后事务B提交了一次修改，事务A又读了一次

  3. 幻读

    幻读主要是前后查询的记录数量不一致

解决这些问题的4种隔离级别

  1. 读未提交 （啥也解决不了）

    一个事务未提交时，变更就可以被其他事务看到

  2. 读提交 （解决脏读）

    一个事务提交了，变更才可以被其他事务看到

    每个select都会生成新的Read View

  3. 可重复读 （解决不可重复读）

    一个事务执行中看到的数据保证和开始时一致（InnoDB 默认的隔离级别）

    事务创建的时候生成一个Read View，整个事务期间都用这个Read View

  4. 串行化 （解决幻读）

    对记录加上读写锁，多个事务对同一记录操作发生读写冲突时，必须等待其他事务执行完成

不建议升级到串行化，效率太低

InnoDB的可重复读级别避免了很多幻读情况

  1. 针对快照读（普通读），用MVCC解决幻读

  2. 针对当前读（select … for update），用next-key lock（记录锁 + 间隙锁）解决幻读

    记录锁：锁一条记录

    间隙锁：锁一个范围，除了记录本身；如：(3, 5)

    next-key lock（临键锁）：记录锁 + 间隙锁，锁一个范围，包括记录本身；如：(3, 5]

MVCC

InnoDB对MVCC的实现依赖：隐藏字段、Read View和undo log

通过隐藏字段和Read View来判断数据的可见性，如果不可见，则找到undo log中的历史版本

每个事务读到的数据版本可能不一样。同一个事务中，只能看到该事务创建Read View之前已经提交的和该事务自己创建的修改
